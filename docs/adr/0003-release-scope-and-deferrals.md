# ADR 0003: 0.20.0 初回リリースのスコープと延期事項

- **ステータス**: 採用
- **日付**: 2026-06-12
- **前提**: [ADR 0001](0001-electron-to-tauri.md)(Tauri 移行)/ [ADR 0002](0002-muyajs-to-muyajs-core.md)(エンジン切替)

## コンテキスト

Tauri 移行・アプリ名変更(Remarks)・@muyajs/core エンジン切替・リリース CI(A2)が
完了し、初回リリース(0.20.0)を出せる状態に近づいた。残っている項目は
「外部依存待ち」「意図的に延期した Electron 版とのパリティギャップ」「戦略課題」に
大別される。何を 0.20.0 に含め、何を含めないかを決めて記録する。

## 決定

### 0.20.0 はこのスコープでリリースする

1. **未署名のままリリースする**(コード署名・公証なし)。
   Apple Developer Program 加入(A1)は外部依存であり、リリースをブロックしない。
   Gatekeeper 警告への対処(`xattr -cr`)をリリースノートに明記する
   (CI のリリースボディに記載済み)。証明書入手後のリリースから署名・公証を有効化する。
2. **下記の延期項目は 0.20.0 に含めない**。いずれも回避策があるか影響が限定的で、
   リリースを遅らせる価値がない。
3. リリース手順は CI(`release-tauri.yml`)に一本化する。タグ前の唯一の手作業は
   `package.json` の `-dev` サフィックス除去。

### 延期項目とその理由

**パリティギャップ(Electron 版にあって Tauri 版にない)**

| 項目 | 延期理由 |
|---|---|
| マルチウィンドウのセッション復元(現状 main のみ) | Tauri は起動時に main しか再生成しない。複数ウィンドウのマニフェスト設計が別途必要。単一ウィンドウ利用が支配的 |
| maximize/fullscreen イベント同期・フォーカス時メニュー再同期 | オーバーレイタイトルバー構成では価値が低い。Tauri に Maximized イベントがなくポーリングが必要 |
| クリップボードからのファイルパス推測 | NSPasteboard のネイティブブリッジが必要。ビットマップ貼り付け(FileReader 経由)は動作する |
| グローバルショートカット | ローカルショートカット(レンダラーディスパッチャ)で主用途は充足 |
| スペルチェックの右クリック候補・カスタム辞書 | NSSpellChecker ブリッジが必要。macOS ネイティブの下線表示は動作する。Win/Linux スペルチェックも mac-first 方針で対象外 |
| ディレクトリ/タブの unwatch | watch はセッション内で単調増加するが有界。レンダラーは stale イベントを no-op する |
| 拡張子別ファイルアイコン(md.icns) | 見た目のみ |
| ripgrep の非 UTF-8 行デコード・前後コンテキスト行 | 検索本体は動作する。エッジケース |

**スコープ外と確定**

- 単一行 `$$...$$` ブロック数式の raw 表示 — Electron 版でも同じ muya パーサー挙動。
- iOS — ADR 0001 の決定のまま。

**戦略課題(リリース後に計画)**

| 項目 | 備考 |
|---|---|
| tauri2.0 → develop の還元 | エンジンが揃い差分は「Tauri 化 + Remarks 化」に縮小済み。実施時は新 ADR を起こす |
| Phase H: `packages/muyajs` の削除 | develop と同期して 0.20.0 リリース後(develop 側の決定に追随) |
| Electron 併走構成の削除 | Phase H とセット |
| tauri-driver 実機 E2E(Linux/Windows CI) | macOS は原理的に不可。Playwright WebKit + モックで代替中 |
| 行コピペ独自仕様の upstream muya への提案 | 維持コスト削減になるなら(ADR 0002 のトレードオフ) |

### 雑務(リリース前後の任意整理)

- マージ済み `tauri2.0-muya-core` ブランチの削除(ローカル + origin)
- エンジン切替の残り QA(図・画像・テーブル・メニュー状態・性能)は
  常用の中で発見次第修正するサイクルとする(重点項目の IME・コピペ・選択は検証済み)

## 検討した代替案

- **署名が揃うまでリリースを待つ** — 外部依存(Apple Developer Program)で時期が
  読めず、未署名でも `xattr` 一回で利用可能。配布を始めてフィードバックを得る価値が勝る。
- **パリティ 100% までリリースを待つ** — 延期項目はいずれも利用頻度・影響が小さく、
  本体(編集体験・1/100 のバンドルサイズ)の価値提供を遅らせる理由にならない。

## 結果

- 0.20.0 のリリース作業は「`-dev` 除去 → タグ push → ドラフト確認 → 公開」に縮小。
- 既知のパリティギャップはこの ADR が利用者向け説明・将来作業の単一参照点になる。
- 延期項目に着手する際は、本 ADR の該当行を更新(または新 ADR を起こして)決定を残す。
